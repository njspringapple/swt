
## 一、需要背诵的核心概念 (Zu merkende Kernkonzepte)

### 1. 系统架构 vs 软件架构 (Systemarchitektur vs Softwarearchitektur)

|概念|德文|中文定义|关注点|
|---|---|---|---|
|**系统架构**|**Systemarchitektur**|软件系统的粗粒度结构 (Grobstruktur)|子系统 (Subsysteme)、硬件分配 (Verteilung auf Hardware)、技术栈 (Technologiestack)、通信协议 (Kommunikationsprotokolle)|
|**软件架构**|**Softwarearchitektur**|单个子系统的内部结构|组件 (Komponenten)、模块关系、架构模式 (Architekturmuster)、架构决策 (Architekturentscheidungen)|

**记忆口诀**：系统架构是"大图" (Big Picture)，包含多个子系统；软件架构聚焦于每个子系统内部。

---

### 2. 4+1 架构视图模型 (4+1 Architektur-Sichten) ⭐必背

|视图|德文|内容|典型工具|
|---|---|---|---|
|**逻辑视图**|**Logische Sicht**|系统中的主要抽象，以对象/实体表示 (Hauptabstraktionen als Objekte/Entitäten)|UML 类图、对象图|
|**过程视图**|**Prozess-Sicht**|运行时的系统进程交互与通信 (Prozesse zur Laufzeit)|活动图、序列图、文本描述|
|**开发视图**|**Entwicklungs-Sicht**|开发者视角的组件划分：包、接口、类 (Pakete, Interfaces, Klassen)|包图、目录结构|
|**物理视图**|**Physische Sicht**|软件组件到硬件的映射：服务器、客户端、网络 (Server, Clients, Netzwerke)|部署图、文本描述|

**判断技巧**：

- 看到**类图 + 实体/业务逻辑** → 逻辑视图
- 看到**活动图 / Workflow** → 过程视图
- 看到**Java 类/接口/包结构** → 开发视图
- 看到**服务器/容器/硬件连接** → 物理视图

---

### 3. 架构决策记录 ADR (Architectural Decision Records) ⭐必背

|术语|德文|中文定义|
|---|---|---|
|**架构决策 (AD)**|**Architectural Decision**|一个有理由的设计选择，针对具有架构重要性的功能或非功能需求|
|**架构重要需求 (ASR)**|**Architecturally Significant Requirement**|对架构和系统质量有可测量影响的需求|
|**架构决策记录 (ADR)**|**Architectural Decision Record**|记录一个 AD 及其理由、权衡与后果的短文档|
|**决策日志**|**Decision Log**|一个项目中所有 ADR 的集合|

**ADR 内容结构**：标题 (Title) → 状态 (Status: Accepted/Deprecated/Superseded) → 上下文 (Context) → 决策 (Decision) → 备选方案 (Alternatives) → 理由 (Rationale) → 后果 (Consequences)

---

### 4. 非功能需求对架构的影响 (NFR → Architekturentscheidungen) ⭐必背

|非功能需求|德文|架构决策|
|---|---|---|
|**性能**|**Leistungsfähigkeit**|时间关键流程不通过网络分发，在少数紧耦合模块中本地执行；通过负载均衡实现可扩展性|
|**信息安全**|**Informationssicherheit**|分层架构，将需保护的内容保存在隔离的层中|
|**可维护性**|**Wartbarkeit**|强解耦、小组件；避免共享数据结构|

---

### 5. 解耦原则 (Entkopplungsprinzipien) ⭐必背

|概念|德文|含义|
|---|---|---|
|**高内聚**|**Hohe Kohäsion**|模块内部元素逻辑上紧密相关 → **好**|
|**低耦合**|**Niedrige Kopplung**|模块之间依赖程度低 → **好**|

**四大解耦手段**：

|手段|德文|说明|
|---|---|---|
|接口作为依赖|Interfaces als Abhängigkeiten|用抽象接口替代具体实现|
|每个系统独立类型|Eigene Datentypen pro System|在系统边界设置 Mapping，外部变更只影响映射层|
|按业务逻辑分离|Trennung nach Fachlichkeit|只在业务相关时复用代码，避免共享工具类|
|避免循环依赖|Keine zyklischen Abhängigkeiten|A→B, B→A 是不允许的|

---

### 6. 七种架构模式 (Sieben Architekturmuster) ⭐必背

|模式|德文|核心特点|
|---|---|---|
|**MVC**|**Model-View-Controller**|Model(数据+业务逻辑)、View(展示)、Controller(处理输入，协调M和V)|
|**分层架构**|**Schichtenarchitektur**|水平分层，每层只能调用其下一层|
|**洋葱架构**|**Onion-Architektur**|外层使用内层，内层只知道接口不知外层具体实现|
|**仓库架构**|**Repository-Architektur**|所有组件共享一个中央数据仓库|
|**Client-Server**|**Client-Server-Architektur**|客户端(UI) ←网络→ 服务器(数据+逻辑)|
|**微服务**|**Microservice-Architektur**|拆分为小型自治服务，通过网络接口通信|
|**管道过滤器**|**Pipes and Filters**|数据依次通过一系列处理步骤（过滤器），输出作为下一步输入|

---

### 7. Thick Client vs Thin Client ⭐必背

|类型|德文|优点|缺点|
|---|---|---|---|
|**Thick Client**|**Thick Client**|少数据传输、可离线工作、本地快速响应、减轻服务器负载|需要更多本地资源、更新维护较复杂|
|**Thin Client**|**Thin Client**|需要更少端设备资源、易于维护（仅需更新服务器）、安全性更好（防止客户端被黑客篡改）、常为可复用标准软件（如浏览器）|依赖网络连接、服务器负载高|

---

### 8. 微服务优缺点 (Vor-/Nachteile Microservices) ⭐必背

**优势 (Vorteile)**：

|德文|中文|
|---|---|
|Unabhängige Skalierbarkeit|独立扩展：资源密集部分可针对性扩展|
|Unabhängige Deployments|独立部署：可快速针对性升级单个组件|
|Technologische Freiheit|技术自由：每个服务可用不同技术实现|
|Fehler-Isolation|故障隔离：一个服务崩溃不导致整个系统崩溃|
|Klare Schnittstellen, kleinere Code-Bases|清晰接口和更小代码库：更好的可维护性|

**劣势 (Nachteile)**：

|德文|中文|
|---|---|
|Zusätzliche Komplexität (Netzwerk-Latenz, Load-Balancing)|额外复杂度（网络延迟、负载均衡）|
|Daten-Konsistenz schwieriger|数据一致性更难保证|
|Operativer Overhead (Deployment, Logging, Monitoring je Service)|运维开销（每个服务都需独立部署/日志/监控）|
|Kommunikations-Kosten (Serialisierung/Deserialisierung)|通信成本（序列化/反序列化）|
|Debugging über mehrere Services hinweg|跨服务调试更复杂|
|Komplizierte Integrations-Tests (Mock-Services)|集成测试复杂（需要 Mock 服务）|

### 9. Pipes and Filters 优缺点

||德文|中文|
|---|---|---|
|**优点**|Wiederverwendbarkeit, Komposition|可复用性、可组合性|
|**缺点**|Daten müssen serialisiert werden; Ausgabe muss mit Eingabe des nächsten kompatibel sein|数据须序列化；一个程序的输出必须与下一个程序的输入兼容|

---

### 10. MVC 通信流程 ⭐必背

用户在 View 中修改数据后的通信流程：

> **用户操作 View** → **View 通知 Controller** → **Controller 更新 Model** → **Model 通知 View（通过 Observer）** → **View 重新显示更新后的数据**

### 11. 设计策略 (Entwurfsstrategien)

|策略|德文|说明|
|---|---|---|
|**分而治之**|**Divide & Conquer**|分解为子系统 → 定义接口 → 独立设计|
|**领域驱动设计**|**Domain-driven Design (DDD)**|统一语言 (Ubiquitous Language)、领域聚焦、Bounded Contexts、独立领域模型|

### 题目2：Thick Client vs Thin Client

**Q:** 比较 Thick Client 和 Thin Client 的优缺点，并各举一个适用的软件例子。  
**Q (DE):** Welche Vor- und Nachteile ergeben sich durch „thick" bzw. „thin" Clients? Nennen Sie je ein Beispiel.

**A:**

**Thick Client（如 Outlook 桌面版、游戏客户端）：**

- ✅ 数据传输少 (weniger Datentransfer)，能更好应对网络延迟或中断
- ✅ 减少服务器计算负载 (weniger Rechenlast auf Server)
- ✅ 允许直接本地响应用户输入 (schnelle lokale Reaktion)
- ❌ 需要更多本地资源、更新维护复杂

**Thin Client（如 Web 浏览器）：**

- ✅ 端设备资源需求少 (weniger Ressourcen auf dem Endgerät)
- ✅ 更易维护——大多数调整只需在服务器端进行 (leichter wartbar)
- ✅ 更安全——用户无法通过篡改客户端获取更多权限 (sicher gegen Client-Hacks)
- ✅ 常为标准软件，可复用于任意应用 (z.B. Browser)
- ❌ 依赖网络连接

### 题目3：Client-Server 基础问题

**Q:** Client-Server 系统是否必须分布在不同的硬件上？Client 和 Server 通过什么介质通信？  
**Q (DE):** Muss ein Client-Server-System immer auf unterschiedliche Hardware verteilt werden? Über welches Medium kommunizieren Client und Server?

**A:**

不一定 (Nein)。Client-Server 系统也可以在同一台计算机上运行。客户端和服务器作为分离组件的逻辑模型保持不变，即使它们物理上在同一设备上运行。

> Das logische Modell von getrennten Client und Server Komponenten bleibt bestehen, auch wenn sie physisch auf demselben Gerät laufen.

Client 和 Server 通过**网络** (Netzwerk) 通信，例如 Internet 或 LAN。

### 题目4：日常架构实例 (Architekturen im Alltag)

**Q:** 请各举一个日常软件的例子，分别符合分层架构和仓库架构。  
**Q (DE):** Nennen Sie jeweils ein Beispiel für Software, die einer Schichtenarchitektur bzw. einer Repository-Architektur folgt.

**A:**

**分层架构 (Schichtenarchitektur)**：操作系统 (Betriebssysteme)、网络协议 (Netzwerkprotokolle)、数据库 (Datenbanken)、Web 应用 (Webanwendungen)

**仓库架构 (Repository-Architektur)**：版本控制系统如 Git (Versionsverwaltungssysteme)、数据库 (Datenbanken)、IDE 如 VSCode 的设置系统

### 题目5：MVC 通信流程

**Q:** 描述用户在 View 中修改数据后，MVC 各组件之间的通信流程。  
**Q (DE):** Beschreiben Sie den Kommunikationsablauf zwischen den Komponenten im MVC-Muster, nachdem ein Nutzer Daten in der View geändert hat.

**A:**

1. 用户在 **View** 中修改数据
2. View 将用户操作通知 **Controller**
3. Controller 处理输入，调用 **Model** 进行更新
4. Model 数据变更后，通过 **Observer 模式** 通知所有注册的 View
5. View 从 Model 获取最新数据并重新显示

> User ändert Daten in View → View informiert Controller → Controller aktualisiert Model → Model benachrichtigt View (Observer) → View aktualisiert Darstellung.

### 题目6：Pipes and Filters

**Q:** Unix 管道 `echo "Hallo, wie geht's?" | wc -c` 体现了什么架构？它有什么优缺点？  
**Q (DE):** Wie nennt sich diese Architektur? Nennen Sie Vor- und Nachteile.

**A:**

- **架构名称**：Pipes and Filters（管道与过滤器）
- **优点**：可复用性 (Wiederverwendbarkeit)、可组合性 (Komposition)——每个程序专注一个任务，可自由组合
- **缺点**：数据必须序列化 (Daten müssen serialisiert werden)；一个程序的输出格式必须与下一个程序的输入格式兼容

### 题目7：微服务适用性判断

**Q:** 一个应用从企业内部订单数据库自动生成发票并通过邮件发送给客户，每天只运行一次，不需要高可用性或可扩展性。为什么不适合使用微服务架构？  
**Q (DE):** Wieso macht es in diesem Fall keinen Sinn, eine Microservice-Architektur zu verwenden?

**A:**

对于这样的应用，微服务架构的复杂性不合理 (nicht gerechtfertigt)。该应用只有少量职责（获取数据、生成发票、发送邮件），这些职责紧密关联。分布式系统带来的通信、运维和复杂度开销远超其收益。

> Für eine solche Anwendung ist die Komplexität einer Microservice-Architektur nicht gerechtfertigt. Der Overhead durch verteilte Systeme, Kommunikation und Betrieb überwiegt den Nutzen.

### 题目8：Pipes and Filters 实践编程

**Q:** 在 CSV 处理管道中，如何添加一个过滤步骤，移除所有非整数的条目？  
**Q (DE):** Welche Komponenten fügen Sie hinzu, um nicht-numerische Einträge zu filtern?

**A:**

新增类 `FilterNonNumeric`，包含 `process(self, input_tokens)` 方法。该方法遍历 token 列表，尝试将每个 token 转为 `int`，成功则保留，`ValueError` 则跳过。

```python
class FilterNonNumeric:
    def __init__(self):
        pass
    def process(self, input_tokens):
        filtered = []
        for token in input_tokens:
            try:
                int(token)
                filtered.append(token)
            except ValueError:
                continue
        return filtered
```

集成到管道中：

```python
pipeline = CsvProcessor()
pipeline.add_step(Tokenize())          # "1,2,5,Nan,7,foo" → ['1','2','5','Nan','7','foo']
pipeline.add_step(FilterNonNumeric())  # → ['1','2','5','7']
pipeline.add_step(Join())              # → "1,2,5,7"
pipeline.run(csv_line)
```

### 题目9：MVC 实现中的正确分离

**Q:** 在 MVC 实现中，如何确保 Model 和 GUI 正确分离？  
**Q (DE):** Wie stellt man eine korrekte Trennung von Model und GUI sicher?

**A（经验法则 / Rule of Thumb）**：

- GUI 和 Model 放在**不同的包** (separate Packages) 中
- Model 中**不应有任何 GUI 代码的 import** (Keine Imports von GUI-Code im Model)
- 使用 **Observer 模式** 进行 Model 到 View 的通信

### 题目10：ADR 概念

**Q:** 什么是 ADR？它包含哪些核心内容？为什么需要 ADR？  
**Q (DE):** Was ist ein ADR? Welche Inhalte hat es? Warum braucht man ADRs?

**A:**

ADR (Architectural Decision Record) 是记录一个架构决策及其理由的短文档。包含：标题 (Title)、状态 (Status)、上下文 (Context)、决策 (Decision)、备选方案 (Alternatives)、理由 (Rationale)、后果 (Consequences)。

ADR 帮助团队理解为什么选择了某个架构方案，以及这个选择带来了哪些权衡和后果。所有 ADR 构成项目的决策日志 (Decision Log)。
