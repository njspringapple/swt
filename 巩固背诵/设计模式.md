
# 🎓 设计模式考前背诵手册 | Entwurfsmuster Klausurvorbereitung

---

## 一、设计模式三大分类总览 | Drei Kategorien

### 1. 创建型 Erzeugungsmuster (Creational Patterns)

> **核心问题**：如何创建对象？ Wie werden Objekte erzeugt?

| 模式                 | 德语            | 英语             | 一句话总结                                |
| ------------------ | ------------- | -------------- | ------------------------------------ |
| **Singleton**      | Einzelstück   | Singleton      | 全局唯一实例，私有构造函数 + static getInstance() |
| **Factory Method** | Fabrikmethode | Factory Method | 让子类决定创建哪种对象，创建与使用分离                  |
| **Builder**        | Erbauer       | Builder        | 链式调用一步步构建复杂对象，解决参数爆炸                 |
| **Prototype**      | Prototyp      | Prototype      | 通过 **clone()** 复制已有对象，避免重复昂贵初始化      |

### 2. 结构型 Strukturmuster (Structural Patterns)

> **核心问题**：如何组合类和对象？ Wie werden Klassen/Objekte zusammengesetzt?

|模式|德语|英语|一句话总结|
|---|---|---|---|
|**Adapter**|Adapter|Adapter|转换接口，让不兼容的类一起工作|
|**Decorator**|Dekorierer|Decorator|层层包装动态增强功能，避免类爆炸|
|**Facade**|Fassade|Facade|为复杂子系统提供统一简化接口|
|**Proxy**|Stellvertreter|Proxy|代理对象控制对真实对象的访问（延迟加载/权限）|
|**Composite**|Kompositum|Composite|树形结构，统一对待叶子和容器（部分-整体）|

### 3. 行为型 Verhaltensmuster (Behavioral Patterns)

> **核心问题**：对象之间如何交互和分配职责？ Wie kommunizieren Objekte?

| 模式                          | 德语                  | 英语             | 一句话总结                              |
| --------------------------- | ------------------- | -------------- | ---------------------------------- |
| **Strategy**                | Strategie           | Strategy       | 算法封装到接口实现类中，运行时可替换                 |
| **Observer**                | Beobachter          | Observer       | 一对多通知机制，Subject 状态变化自动通知 Observers |
| **Iterator**                | Iterator            | Iterator       | 统一遍历接口 hasNext()/next()，隐藏内部数据结构   |
| **State**                   | Zustand             | State          | 每种状态一个类，行为委托给当前状态对象                |
| **Visitor**                 | Besucher            | Visitor        | 双重分派，在不修改元素类前提下新增操作                |
| **Memento**                 | Memento             | Memento        | 保存/恢复对象状态（Undo），不破坏封装性             |
| **Chain of Responsibility** | Zuständigkeitskette | Chain of Resp. | 请求沿处理者链传递，直到被处理                    |
| **Mediator**                | Vermittler          | Mediator       | 中介者协调多对多通信，避免网状耦合                  |
| **Interpreter**             | Interpreter         | Interpreter    | 为语法规则定义表达式类，构建语法树递归求值              |

## 二、考试重点：每种模式详细背诵卡 | Lernkarten

---

### 🔵 Singleton 单例模式 | Einzelstück

**适用场合 / Anwendung：**  
配置管理器 (ConfigManager)、日志记录器 (Logger)、数据库连接池——系统中某个类只能有一个实例的场景。

**三要素（必背）：**

1. **private 构造函数** → 禁止外部 new
2. **private static 实例字段** → 唯一实例
3. **public static getInstance() 方法** → 全局访问点，懒加载

**代码识别特征（考试原题）：**

```java
public class Config {
    private static Config instance;
    private Config() {}  // 私有构造！
    public static Config getInstance() {
        if (instance == null) {
            instance = new Config();
        }
        return instance;
    }
}
// → 这是 Singleton Pattern
```

**UML 关键点：** 类自引用箭头 `<<creates>>` 指向自身，构造函数标 `-`（private），getInstance() 标 `+` 和 `{static}`。

---

### 🔵 Factory Method 工厂方法 | Fabrikmethode

**适用场合 / Anwendung：**  
创建对象的类型需要在运行时决定；想把 `new ConcreteClass()` 从业务逻辑中分离出来；产品有共同接口但实现不同。

**核心结构（4 个角色）：**

1. **Product (接口)** — 产品统一接口，如 `Transport`
2. **Concrete Product** — 具体产品，如 `Truck`, `Ship`
3. **Creator (抽象类)** — 定义 `abstract createProduct()` + 业务模板方法
4. **Concrete Creator** — 实现工厂方法，如 `RoadLogistics` 返回 `new Truck()`

**与 Singleton 常见组合：** Singleton + Factory Method（工厂本身为单例）

**UML 关键点：** Creator 和 Product 两棵平行的继承树，Creator 用虚线箭头 `<<creates>>` 指向 Product。

---

### 🔵 Builder 建造者模式 | Erbauer

**适用场合 / Anwendung：**  
构造函数参数太多（Telescoping Constructor）；很多参数是可选的；构建步骤有顺序要求。

**核心特征：**

1. **Builder 类的每个 setter 返回 `this`** → 链式调用 (Fluent API)
2. **必填参数在 Builder 构造函数中**，可选参数有默认值
3. **最终调用 `build()` 返回产品对象**
4. 产品类的构造函数通常是 **private**，只有 Builder 能调用

**代码使用形式（必背）：**

```java
Computer pc = new Computer.Builder("i7", 32)
    .storage("1TB")
    .gpu("RTX 4090")
    .build();
```

---

### 🔵 Prototype 原型模式 | Prototyp

**适用场合 / Anwendung：**  
创建对象成本高（如需读磁盘/数据库）；需要大量相似对象；想避免重复初始化。

**核心特征：**

1. **clone() 方法**通过拷贝构造函数创建新对象
2. 注意 **深拷贝 vs 浅拷贝** (Tiefe vs. flache Kopie)
3. 可选 **PrototypeRegistry** 管理多种原型

---

### 🟢 Adapter 适配器模式 | Adapter

**适用场合 / Anwendung：**  
已有类的接口与系统期望的接口不匹配（特别是第三方库）；不想/不能修改两端代码。

**核心结构（3 个角色）：**

1. **Target (接口)** — 客户端期望的接口，如 `Logger`
2. **Adaptee** — 已有的不兼容类，如 `FancyThirdPartyLogger`
3. **Adapter** — 实现 Target 接口 + 持有 Adaptee 引用 → 在方法中转换调用

**UML 关键点：** Adapter 同时"实现 Target"和"关联 Adaptee"。

---

### 🟢 Decorator 装饰器模式 | Dekorierer

**适用场合 / Anwendung：**  
需要动态组合功能而不想产生类爆炸 (Klassenexplosion)；功能可以自由叠加。

**考试原题场景——窗口系统：**

> 窗口可以有 Titelleiste / Statusleiste / ScrollBar 等，Decorator 避免为每种组合写子类。

**核心结构（4 个角色）：**

1. **Component (接口)** — 如 `Window` 或 `Beverage`
2. **Concrete Component** — 如 `SimpleWindow`
3. **Decorator (抽象)** — 实现 Component + 持有 Component 引用
4. **Concrete Decorator** — 如 `TitleBarDecorator`, `ScrollBarDecorator`

**关键特征：**

- Decorator **实现与被装饰者相同的接口**
- Decorator **持有被装饰者的引用**
- 方法中**先调用被装饰者的方法，再增强**
- 可以**像套娃一样多层嵌套**

**UML 关键点：** Decorator 既 `implements Component` 又 `has-a Component`（自关联箭头）。

---

### 🟢 Facade 外观模式 | Fassade

**适用场合 / Anwendung：**  
客户端需要与多个子系统交互，操作步骤复杂且有固定顺序；想为常见操作提供简化入口。

**核心特征：**

- Facade 类封装多个子系统的调用，提供简化的高层方法
- **不隐藏子系统**，需要细粒度控制时仍可直接调用子系统
- 减少客户端与子系统的耦合

---

### 🟢 Proxy 代理模式 | Stellvertreter

**适用场合 / Anwendung：**  
需要延迟加载昂贵资源 (Virtual Proxy)；需要访问权限控制 (Protection Proxy)；需要远程调用的本地代表 (Remote Proxy)。

**核心特征：**

- Proxy 与 Real Subject 实现**同一个接口**
- 客户端不知道用的是 Proxy 还是 Real Subject
- 与 Decorator 区别：Proxy **控制访问**，Decorator **增强功能**

---

### 🟢 Composite 组合模式 | Kompositum

**适用场合 / Anwendung：**  
树形结构（部分-整体关系）；希望客户端统一对待叶子节点和容器节点。

**考试原题场景——数学表达式（Term）：**

> Term 递归定义：Literal 是 Term，两个 Term 做二元运算也是 Term。用 Composite Pattern 建模。

**核心结构（3 个角色）：**

1. **Component (接口/抽象)** — 如 `Term`，方法 `evaluate(): int`
2. **Leaf** — 如 `Literal`（叶子，存值）
3. **Composite** — 如 `BinaryOp`（容器，含 left/right 两个 Term 子节点）

**考试 UML 结构（原题答案）：**

```
        Term (abstract/interface)
        + evaluate(): int
       /      |        \
  Literal  Parenthesized  BinaryOp
                          /   |   \
                   Addition  Subtraction  Multiplication  Division
```

- BinaryOp 有 `left: Term` 和 `right: Term` 两个子引用
- Parenthesized 有 `term: Term` 一个子引用

**常见组合：** Composite + Visitor, Composite + Iterator

---

### 🔴 Strategy 策略模式 | Strategie

**适用场合 / Anwendung：**  
同一操作有多种算法/策略；需要运行时替换算法；消除大段 if-else。

**核心结构（3 个角色）：**

1. **Strategy (接口)** — 如 `PaymentStrategy`，方法 `pay(amount)`
2. **Concrete Strategy** — 如 `CashPayment`, `CreditCardPayment`
3. **Context** — 如 `CashRegister`，持有 Strategy 引用，委托调用

**考试原题：画 Strategy 的 UML（Aufgabe 1c）**

```
Context ───────> <<interface>> Strategy
                     + algorithm(): void
                    /          \
         ConcreteStrategyA   ConcreteStrategyB
```

- Context 有属性 `- strategy: Strategy`
- Context 有方法 `+ setStrategy(s: Strategy): void` 和 `+ execute(): void`
- Context → Strategy 是关联（实线箭头）
- ConcreteStrategy → Strategy 是实现（虚线三角箭头）

---

### 🔴 Observer 观察者模式 | Beobachter

**适用场合 / Anwendung：**  
一个对象状态变化需要通知多个依赖者；需要动态增减订阅者；事件驱动系统。

**核心结构（考试原题要求画 UML）：**

1. **Observer (接口)** — 方法 `update()`
2. **Subject (类/接口)** — 方法 `subscribe(o)`, `unsubscribe(o)`, `notify()`；属性 `observers: List<Observer>`
3. **Concrete Observer** — 实现 `update()`
4. Subject → Observer 是关联（0..*）

**UML 必背：**

```
Subject                      <<interface>> Observer
- observers: List<Observer>   + update(): void
+ subscribe(o): void              |
+ unsubscribe(o): void      ConcreteObserver
+ notifyAll(): void          + update(): void
        |
        └── 0..* ──────>  Observer
```

---

### 🔴 State 状态模式 | Zustand

**适用场合 / Anwendung：**  
对象行为随内部状态变化（状态机）；消除状态相关的 if-else 嵌套。

**与 Strategy 的对比（考试常考）：**

| Strategy | State       |              |
| -------- | ----------- | ------------ |
| 切换者      | **客户端**主动选择 | **对象自身**自动切换 |
| 目的       | 替换算法        | 改变行为随状态      |
| 结构       | 几乎相同        | 几乎相同         |

---

### 🔴 Visitor 访问者模式 | Besucher

**适用场合 / Anwendung：**  
需要对一组对象执行多种不同操作；不想修改元素类来新增操作。

**核心机制——双重分派 (Double Dispatch)：**

1. `element.accept(visitor)` → 第一次分派：根据元素类型
2. `visitor.visitXxx(this)` → 第二次分派：根据访问者类型

**常见组合：** Composite + Visitor

---

### 🔴 Memento 备忘录模式 | Memento

**适用场合 / Anwendung：**  
Undo/Redo 功能；需要保存和恢复对象状态；不想破坏封装性。

**考试原题场景——Mensa Speiseplan：**

**三个角色（必背）：**

1. **Originator（原发器）= Speiseplan** — 创建 Memento + 从 Memento 恢复状态
    - `save(): SpeiseplanSnapshot`
    - `restore(snapshot): void`
2. **Memento（备忘录）= SpeiseplanSnapshot** — 存储状态的不可变对象
    - 包含所有字段的副本 + savedAt: DateTime
    - 提供 getter（不提供 setter！）
3. **Caretaker（管理者）= SpeiseplanManager** — 管理 Memento 集合
    - `save(plan)` / `restore(plan, date)`
    - **不知道 Memento 内部细节**

**差的设计（考试必知）为什么差：**

- 所有字段 public → 违反 Encapsulation
- SpeiseplanManager 必须知道 Speiseplan 的内部结构
- 未来字段变化 → Manager 也要改 → 维护噩梦

**好的设计关键：** Originator 自己负责创建和恢复 Snapshot，Caretaker 只管存储。

---

### 🔴 Iterator 迭代器模式 | Iterator

**适用场合 / Anwendung：**  
遍历集合时不想暴露内部数据结构；统一不同数据结构的遍历方式。

**核心接口：** `hasNext(): boolean` + `next(): Element`

---

### 🔴 Chain of Responsibility 责任链 | Zuständigkeitskette

**适用场合 / Anwendung：**  
多个处理者可能处理同一请求；处理者的顺序/数量需要动态调整。

**核心特征：**

- 每个 Handler 有 `nextHandler` 引用 → 链式结构
- 不能处理就 `passToNext()`

---

### 🔴 Mediator 中介者模式 | Vermittler

**适用场合 / Anwendung：**  
对象间网状耦合严重；需要集中管理多对多通信。

**与 Observer 的区别（常考）：**

|Observer|Mediator|
|---|---|---|
|关系|一对多|多对多|
|方向|Subject → Observers|Colleagues ↔ Mediator|

---

## 三、模式对比速查表（考试高频考点）| Vergleich

|对比项|模式A|模式B|关键区别|
|---|---|---|---|
|Strategy vs State|Strategy|State|Strategy: 客户端选择算法；State: 对象自动切换行为|
|Decorator vs Proxy|Decorator|Proxy|Decorator: 增强功能(可多层)；Proxy: 控制访问(一层)|
|Composite vs Decorator|Composite|Decorator|Composite: 部分-整体统一接口；Decorator: 动态添加职责|
|Observer vs Mediator|Observer|Mediator|Observer: 1→N通知；Mediator: N↔N协调|
|Factory Method vs Builder|Factory Method|Builder|FM: 创建哪种产品；Builder: 如何一步步构建复杂产品|
|Adapter vs Facade|Adapter|Facade|Adapter: 转换1个接口；Facade: 简化多个子系统的入口|

## 四、常见组合 | Häufige Kombinationen

1. **Singleton + Factory Method** — 工厂本身是单例
2. **Composite + Visitor** — 对树结构的节点执行不同操作
3. **Composite + Iterator** — 统一遍历树结构

## 七、UML 画图速记要点 | UML Tipps

1. **接口 (Interface)** — 类名上方写 `<<interface>>`，方法名用斜体或正常
2. **抽象类** — 类名用 _斜体_
3. **实现关系 (implements)** — **虚线 + 空心三角** 从子类指向接口
4. **继承关系 (extends)** — **实线 + 空心三角** 从子类指向父类
5. **关联 (Association)** — **实线箭头**，可标多重性(0..*, 1)和角色名
6. **依赖 (Dependency)** — **虚线箭头**，常标 `<<creates>>` 或 `<<uses>>`
7. **可见性符号**：`+` public, `-` private, `#` protected
8. **static** 用下划线或标注 `{static}`

### 题目 6：Memento 分析题

以下是一个 Speiseplan 系统的代码。请解释为什么这是一个糟糕的设计：

```java
class Speiseplan {
    public Gericht montag;
    public Gericht dienstag;
```

### 题目 6 答案

这个设计有以下问题：

1. **违反封装原则 (Encapsulation)**：Speiseplan 的所有字段都是 public，内部结构完全暴露。
2. **外部类了解内部结构**：SpeiseplanManager 必须知道 Speiseplan 有哪些字段以及它们的名字，才能逐个赋值进行保存和恢复。
3. **维护困难**：如果 Speiseplan 增加新字段（如 `samstag`）或修改字段名，SpeiseplanManager 也必须同步修改。
4. **保存/恢复逻辑分散**：保存和恢复的责任不在 Speiseplan 自身，而是由外部 Manager 操作，逻辑散布在多个类中。
5. **一致性无保证**：没有机制确保 Snapshot 和 Speiseplan 的字段始终保持同步。

```
             |   (abstract/     |                     │
```

### 题目 11 答案

**相似点：** 两者的 UML 结构几乎相同——都有一个 Context 类持有一个接口引用，接口有多个实现类。Context 的行为委托给该接口对象。

**核心区别：**

- **State**：由**对象自身**在方法执行过程中自动切换状态（`setState()`在 ConcreteState 内部调用）。状态之间有转换关系，行为随状态变化。

### 题目 12 答案

1. **Singleton + Factory Method**：工厂类本身设计为单例，确保全局只有一个工厂实例负责创建产品对象，避免重复创建工厂。
    
2. **Composite + Visitor**：Composite 构建了树形数据结构（如 AST 语法树），Visitor 可以在不修改树节点类的前提下，为树定义新的遍历操作（如求值、打印、类型检查等）。
    
3. **Composite + Iterator**：Composite 的树形结构需要统一遍历，Iterator 提供了统一的遍历接口而不暴露树的内部结构。

### 题目 15 答案

**a) 错误。** Singleton 的构造函数必须是 **private** 的，以防止外部通过 new 创建多个实例。

**b) 正确。** 这是 Decorator 模式的核心特征之一——Decorator 实现与被装饰对象相同的接口（Component），并持有该接口的引用。

**c) 错误。** 在正确的 Memento 实现中，Caretaker **不需要**了解 Originator 的内部字段。Caretaker 只通过 Memento 的公开接口（如 getDate()）管理 Memento 对象，保存/恢复逻辑由 Originator 自己完成。

**d) 错误。** Observer 模式中，Observer 不主动轮询。而是 Subject 状态变化时**主动通知**所有已注册的 Observer（调用 update()）——这是**推送 (Push)** 模式，不是轮询 (Polling)。

**e) 正确。** 这是 Composite 模式的核心思想——叶子节点（Leaf）和容器节点（Composite）都实现同一个 Component 接口，使客户端可以统一对待两者。
